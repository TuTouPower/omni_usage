# 代码审阅报告：2026-06-14 ~ 2026-06-15

## 一、高严重度问题

### 1. `get_raw` 仍泄露 HTTP body（d1dd131 — 与 e57e198 修复矛盾）

- **问题**：`probe-executor.ts` 的 `get_raw` 方法在 HTTP 4xx/5xx 时仍将 `body_snippet`（200 字符）包含在 Error message 中，而 e57e198 已对 `get_json` 做了 body 脱敏。两者不一致，body 可能包含敏感的服务端错误详情。
- **建议**：`get_raw` 的错误处理应与 `get_json` 一致，不泄露 body 内容，改为 `log.debug` 记录。

### 2. `files.list` 符号链接遍历风险（1bf166d）

- **问题**：`is_within_allowed` 使用前缀匹配虽然处理了 `..`，但未处理符号链接。恶意 connector 可在允许目录内创建指向外部目录的符号链接，`list_dir_recursive` 会跟随符号链接遍历系统任意文件。且 `list_dir_recursive` 无深度限制，循环符号链接可导致栈溢出。
- **建议**：(1) 在 `list_dir_recursive` 中使用 `lstat` 检测符号链接并跳过；(2) 添加递归深度限制（如 `maxDepth=10`）。

### 3. ABI 切换脚本的进程清理不安全（370030e）

- **问题**：`ensure_electron_abi.mjs` 使用 PowerShell 通配符匹配 `OmniUsage` 进程名强制终止，在 CI 环境可能误杀其他实例。`execSync` 中拼接的 PowerShell 命令虽然使用转义引号，但多层转义在复杂路径环境下可能被绕过。
- **建议**：(1) 进程清理改用 PID 文件追踪，避免通配符杀进程；(2) 添加超时保护防止 rebuild 挂起。

### 4. `detect_metric_type` 的 `remaining` → `used` 语义映射反转（d1dd131）

- **问题**：`x-ratelimit-remaining` 实际是"剩余量"而非"已用量"，但代码将其映射为 `used`。后续 `ratio = used / limit` 的阈值判断（0.9 = critical）实际上是基于剩余比例而非使用比例，**逻辑反了**——remaining 越小说明使用越多，但代码认为 used 越大越 critical，而 remaining 被当作 used。
- **建议**：修正语义映射，应计算 `used = limit - remaining`。

## 二、中严重度问题

### 5. `/auth/i` redaction 模式过于宽泛（4d901a2）

- **问题**：`SECRET_KEY_PATTERNS` 中 `/auth/i` 会匹配 `author`、`authorization_url` 等非敏感字段名，误脱敏正常配置项。`/private/i` 同理可能匹配 `private_endpoint` 等网络配置。
- **建议**：改为 `/^auth$|auth_token|auth_key/i` 或增加负向前瞻/白名单排除机制。

### 6. SessionManager IPC 缺少 URL 校验，存在 SSRF 风险（7cf9ff3）

- **问题**：`SessionLoginRequest.login_url` 未验证是否为合法 HTTPS URL。恶意渲染进程可传入 `file:///` 或内部网络地址，利用主进程发起 SSRF。`on_before_send_headers` 未限制 URL pattern，登录窗口导航到第三方时请求头会泄露。
- **建议**：(1) 添加 URL 协议白名单（仅 https）；(2) 对 `onBeforeSendHeaders` 限制 URL pattern；(3) 使用错误码代替字符串匹配（`message.includes("already in progress")` 太脆弱）。

### 7. Logger scrubber 内存只增不减（9dea195）

- **问题**：`registered_values` 是 `Set<string>`，vault 值被删除后不会自动 unregister。长期运行可能导致内存缓慢增长，正则越来越长影响性能。
- **建议**：增加 scrubber 定期清理机制或在 `vault.delete` 时调用 `unregister`；对注册值数量设上限（如 10000）。

### 8. 代理 URL 未做格式校验（a24ba9d）

- **问题**：用户输入任意字符串直接传给 `ProxyAgent`，可能触发未预期的解析行为或异常。如果配置 `http://user:pass@proxy:8080` 格式，密码会存入 config.json 明文。
- **建议**：输入框添加 URL 格式验证（至少验证协议为 http/https/socks）；文档提示代理密码的安全风险。

### 9. VendorMark color 通过正则替换 SVG stroke（7148f75）

- **问题**：`replace(/stroke="[^"]*"/, ...)` 只替换第一个匹配，多 stroke 路径的 SVG 会遗漏；且替换 stroke 不影响 fill，部分图标效果不符预期。
- **建议**：用 CSS `color` + `currentColor` 机制控制颜色，而非字符串替换。

### 10. 折叠状态持久化队列缺少错误恢复机制（24d1fd9）

- **问题**：`useRef` 缓存前值 + `JSON.stringify` 比较防重写逻辑较复杂，`save_queue_ref` 串行化实现需确保队列不会卡死。
- **建议**：添加超时保护或错误处理，防止 save_queue 卡住。

### 11. CPA connector 大量逻辑集中在一个文件（c402787）

- **问题**：`connectors/cpa/connector.ts` 新增 459 行，5 个 provider 的 fetch+parse 全部在一个文件中，可维护性差。
- **建议**：按 provider 拆分为独立模块，主 connector 文件只做调度。

## 三、低严重度问题

### 12. CpaCard key 使用 label 而非 account_id（c5e19f3）

- **问题**：`key={`${row.provider}-${row.account_label}`}` 用 label 做 key，同名账号会冲突。
- **建议**：key 改为 `` `${row.provider}-${row.account_id}` ``。

### 13. ADD_COMMON_SERVICES 重复定义（ae0c3fa）

- **问题**：在 `SettingsView` 和 `AddAccountDialog` 中各定义一份，存在重复。
- **建议**：提取到共享模块统一导出。

### 14. 密钥遮罩固定 20 字符（6fbc376）

- **问题**：`"•".repeat(20)` 固定长度不反映真实密钥长度，可能误导用户。
- **建议**：考虑 `"•".repeat(Math.min(value.length, 20))` 保留长度信息。

### 15. ConfirmDelete 蒙层用 onMouseDown 关闭（6aff03b）

- **问题**：`onMouseDown` 关闭蒙层可能与内部拖拽交互冲突。
- **建议**：蒙层关闭改用 `onClick` 更安全。

### 16. Tavily 子项 limit 语义不准确（22d2c5a）

- **问题**：搜索/爬取等子项的 limit 设为 `plan_usage`（已用总量）而非 `plan_limit`（总量上限）。
- **建议**：子项 limit 应使用 `plan_limit` 或 `null`。

### 17. GLM 时间戳启发式判断（a761202）

- **问题**：`raw > 1e10` 区分秒/毫秒级时间戳不完全可靠。
- **建议**：可接受，属常见做法。

### 18. MiMo 硬编码浏览器 headers（cf67435）

- **问题**：User-Agent/Referer/Origin 硬编码，平台更新反爬策略需同步修改。
- **建议**：无严重问题，可后续优化。

### 19. Vault 全局互斥锁降级并发性（e57e198）

- **问题**：从 per-key 锁改为全局互斥锁，不同 key 串行化。对低频写入场景可接受。
- **建议**：如后续 vault 写入频繁可考虑恢复 per-key 粒度锁。

### 20. `window.open` 未检查返回值（a7bd9a0）

- **问题**：弹窗拦截器可能静默失败。
- **建议**：可接受，属于渐进式增强。

## 六、整体评价

**安全加固方向正确**：错误消息脱敏、Logger scrubber 自动脱敏、provider 白名单、CPA 强制显式 endpoint 都是优秀的安全实践。

**重构质量高**：术语统一（UsageItem→MetricRecord、PluginConfiguration→ConnectorConfiguration）保留了 deprecated 别名保证向后兼容，测试同步到位。

**Connector 实现整体规范**：类型安全和测试覆盖良好，但 CPA connector 应拆分。

**优先修复建议**：

1. 先修 4 个高严重度问题，尤其是 `get_raw` body 泄露（安全回归）和 `remaining/used` 语义反转（逻辑错误）
2. 再处理 SSRF 和 redaction 模式过宽等中严重度问题
3. 低严重度问题可排入后续迭代
